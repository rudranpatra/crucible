I built an engine that runs 6 adversarial agents against your CI/CD pipeline. Not a linter. Not a rules engine. Agents. They spawn, attack, score themselves on effectiveness, die when they underperform, and evolve between runs. Here's what I actually shipped and what it took to get there honest. ── What Crucible does: Six attack agents run concurrently against your GitHub Actions or Playwright pipeline: • TimingAgent — injects delay profiles (race windows, cascade delays, timeout probes) • EnvCorruptionAgent — null injection, overflow, path traversal, SQL injection patterns on env vars • StepReorderAgent — shuffles execution order to expose hidden dependency assumptions • NetworkChaosAgent — packet loss, DNS flap, connection reset, latency spikes, partial responses • DependencyDriftAgent — yanked versions, unpinned packages, transitive conflicts, major bumps • SupplyChainAgent — static audit: unpinned action SHA, pull_request_target + checkout exploit, script injection via github.event, overpermissive GITHUB_TOKEN Each run produces a 0–100 resilience score broken into four components: survival rate, blast containment, recovery speed, attack surface coverage. Every failure is saved as a replayable trace. Same seed → same mutations → reproducible results. ── What I had to fix before I could ship this honestly: The first version had a bug I almost shipped: the resilience score's "recovery speed" component (20% of the total score) was derived from random.uniform(). The score looked real. It wasn't. Scores are now computed from actual mutation parameters — delay magnitude, chaos profile, drift severity. Shadow agents had a "promotion" system that counted wins but never fed them back. When a shadow consistently outperformed production, it incremented a counter. The winning perturbation config was thrown away. That's fixed — winning configs are now stored and exploited on subsequent runs. The evolutionary feedback loop actually closes. DarwinScorer and TraceMemory wrote JSON files with no locking. Concurrent pipeline runs would silently corrupt state. Fixed with fcntl exclusive locks and atomic rename on every write. The score trend tracker had the comparison backwards. "Improving" fired when scores went down. ── What's still a model, not injection: The attacks analyze your pipeline YAML and simulate failure conditions. They don't inject chaos into live runner processes. That distinction matters and I'll say it plainly: A "timing" failure today means: Crucible determined your step timeout is tighter than our delay profile. Not: we actually delayed your process and watched it die. That closes in v1: actual process-level fault injection against sandboxed runner execution. ── The supply chain piece is what I think matters most right now: GitHub Actions is the biggest underaudited attack surface in modern engineering orgs. Unpinned actions (actions/checkout@v4 instead of a commit SHA) mean a compromised upstream repo silently becomes RCE in your pipeline. pull_request_target combined with a PR head checkout runs untrusted contributor code with write-scoped GITHUB_TOKEN. User-controlled github.event values interpolated into run: blocks are script injection. Missing permissions blocks leave GITHUB_TOKEN at default scope — broad write on most event types. These aren't theoretical. They're the pattern behind recent high-profile pipeline compromises. Crucible now checks all of them on every run. ── Try it: pip install crucible-gym crucible attack .github/workflows/ci.yml If you run GitHub Actions at scale: the supply_chain check alone is worth running. #DevSecOps #PlatformSecurity #ChaosEngineering #GitHubActions #SupplyChain #OpenSource
